# Iterative Review Fix

## Overview

Превращать набор замечаний в управляемый контур исправления, а не в хаотичную серию точечных правок.

Доводить change set (набор изменений) до состояния, в котором открытых findings (замечаний) не остаётся вовсе.

Держать ответственность разделённой:

- агент отвечает за проверку замечаний по фактам, порядок исправлений, реализацию, качество и повторное ревью;
- пользователь отвечает за product choices (продуктовые выборы), допустимые компромиссы поведения, scope (границы) исправлений и осознанные отложенные решения.

Любой вопрос, который влияет на бизнес-смысл поведения, состав выдачи, пользовательский сценарий, внешний контракт, совместимость или допустимый риск выпуска, обязан быть задан пользователю.

Сохранять след работы в компактном JSON-first формате. Источник истины должен читаться и моделью, и автоматикой без дополнительного парсинга Markdown.

## Artifact Workspace

Создавать отдельную директорию под каждый прогон навыка.

По умолчанию хранить артефакты в текущем work root (рабочем корне) по пути:

```text
.codex-artifacts/iterative-review-fix/sessions/<timestamp>-<task-slug>/
```

Для инициализации использовать скрипт:

```bash
python3 "${CODEX_HOME:-$HOME/.codex}/skills/iterative-review-fix/scripts/init_review_fix_run.py" \
  --workspace-root <ABSOLUTE_WORK_ROOT> \
  --task "<RAW_USER_TASK>" \
  --slug "<SHORT_ASCII_SLUG>" \
  --base-branch "<BASE_BRANCH>" \
  --entry-mode "<auto|initial-diff-review|findings-fix-cycle>"
```

Скрипт создаёт компактный набор артефактов:

- `manifest.json` как машинно-читаемую карту сессии и схемы;
- `session-state.json` как SSOT текущего состояния review-fix цикла;
- `event-log.jsonl` как append-only журнал фактов и решений;
- `reviews/iteration-N.json` как отдельный JSON-документ на каждый review pass.

Структуру и назначение смотреть в [references/artifact-layout.md](references/artifact-layout.md).

`session-state.json` считать главным рабочим документом. Если поле расходится с устаревшей заметкой агента, верить `session-state.json`.

## Workflow

1. Определять entry mode (режим входа) до любых правок.
- Если working tree (рабочее дерево) чистое и пользователь просит ревью ветки, PR или change set относительно base branch, входить в `initial-diff-review`.
- В `initial-diff-review` первый проход обязан быть ревью текущего diff относительно указанной или надёжно выведенной base branch.
- В `initial-diff-review` до первого review pass запрещено вносить кодовые правки, расширять change set, запускать fix-oriented verification (проверки под пакет исправлений) и подменять исходный diff уже изменённым состоянием ветки.
- Если пользователь уже принёс findings, comments или локальный change set на доработку, входить в `findings-fix-cycle`.
- Если base branch не указана явно, использовать только однозначную базовую ветку из локального контекста. Если однозначности нет, задать пользователю короткий вопрос и не начинать правки до ответа.
- Сразу после выбора режима записывать его в `manifest.json`, `session-state.json` и `event-log.jsonl`.

2. Собирать контекст для выбранного режима.
- Прочитать локальный `AGENTS.md` и ближайшие проектные правила, если работа идёт внутри репозитория.
- Найти исходные review findings (замечания), связанный diff (дифф), ветку, код, тесты, миграции и затронутые контракты.
- Сразу после старта создать artifact workspace (рабочее пространство артефактов).
- Сохранить исходную задачу, сырой список замечаний, ограничения и проектный контекст внутрь `session-state.json`.

3. Если выбран `initial-diff-review`, проводить первый review pass до первого исправления.
- Запускать свежего сильного субагента на критическое ревью diff текущей ветки относительно base branch.
- До запуска субагента подготовить `reviews/iteration-1.json` с basis (основанием) прохода, review surface (поверхностью ревью), findings in scope (замечаниями в контуре) и prompt summary (сводкой промпта).
- Если первый проход не нашёл ни одного замечания и не поднял business questions (бизнес-вопросы), завершать цикл без правок как review-only result (результат только ревью).
- Если замечания найдены, переводить их в рабочий backlog (список работы) и только после этого планировать fix batch.

4. Разобрать замечания по фактам.
- Для каждого замечания определить: подтверждается ли оно, частично ли подтверждается, уже ли исправлено, или требует уточнения.
- Не исправлять вслепую только по формулировке ревью; сначала сверять замечание с кодом и локальными правилами.
- Для каждого finding (замечания), которое меняет внешне наблюдаемое поведение существующего URL, списка, истории, отчёта или выборки, отдельно фиксировать behavior delta (дельту поведения): кто видит данные, какие записи попадают в выдачу, какие поля приходят, как считается total/count.
- Сохранять разбор, стратегию исправления, changed surface (затронутую поверхность) и текущий implementation plan (план выполнения) в `session-state.json`.

5. Отделять рабочие исправления от бизнес-развилок.
- Считать бизнес-развилкой изменение, которое меняет поведение продукта, внешний контракт, допустимый компромисс качества, совместимость или состав поставки.
- Считать бизнес-развилкой любое сужение или расширение состава выдачи на существующем контракте: фильтрацию записей, скрытие операций, изменение visibility rules (правил видимости), narrowing by type/status/reference/public marker (сужение по типу, статусу, связи или публичному признаку), изменение total/count и любых правил попадания записи в результат.
- Считать бизнес-развилкой даже тот случай, когда внутренний документ, task-note или локальный SSOT рекомендует скрывать часть данных, но пользователь явно не подтверждал изменение поведения существующего публичного или пользовательского сценария в рамках текущего прогона.
- Любую подтверждённую бизнес-развилку считать закрытой только после явного ответа пользователя в текущем прогоне.
- Поднимать бизнес-развилки пользователю вопросом и сохранять их в `session-state.json` и `event-log.jsonl`.

6. Выполнять fix batch (пакет исправлений).
- Сначала убирать подтверждённые дефекты, которые дают максимальный риск для поведения, архитектуры и качества.
- Следовать локальным правилам по инспекциям, quality gate (контролю качества) и тестам.
- Если проект требует сначала исправить inspections (инспекции), делать это до quality и тестов.
- Обновлять `session-state.json` после каждого meaningful fix batch (содержательного пакета исправлений).

7. Проводить до трёх review passes (проходов ревью) суммарно.
- Считать `initial-diff-review` первым проходом, если он применялся.
- После meaningful fix batch всегда проводить новый review pass.
- Всего на один прогон навыка допускается не более трёх review passes, включая initial diff review и последующие re-review.
- На каждой итерации запускать нового сильного субагента.
- Всегда использовать `fork_context=false`.
- Предпочитать сильную модель уровня `gpt-5.4` с `reasoning_effort=high`, если она доступна.
- Передавать субагенту только локально нужный контекст: подтверждённые замечания, что уже исправлено, текущий diff, результаты верификации и открытые вопросы.
- Если в сессии субагента доступен skill `$final-gate-review`, прямо требовать использовать его как review stance (режим ревью) внутри текущего re-review.

8. Сохранять результат review pass до любых новых правок.
- Сразу после `wait_agent` сохранить raw response (сырой ответ) субагента в `reviews/iteration-N.json`.
- В этом же checkpoint (контрольной точке) нормализовать findings, regression risks (риски регрессии), questions for user (вопросы пользователю), next fix batch (следующий пакет исправлений), what was checked (что проверено) и exit decision (решение о продолжении).
- Для записи checkpoint использовать скрипт `scripts/persist_review_pass.py`, а не ручное переписывание JSON.
- Добавить событие в `event-log.jsonl`.
- Только после записи этих данных разрешены новые кодовые правки, дополнительные проверки и следующий pass.
- Если этот checkpoint не записан, iteration (итерация) считается незавершённой даже при наличии ответа субагента в UI.

9. Возвращать финальный результат.
- Показывать, какие замечания закрыты, какие остались и почему.
- Отдельно перечислять бизнес-решения, подтверждённые пользователем по ходу цикла.
- Отдельно помечать остаточный риск и открытые вопросы.
- Объявлять результат merge-ready только если открытых findings не осталось совсем.
- Сохранять финальный статус в `session-state.json` и дописывать завершающее событие в `event-log.jsonl`.

## Up-To-Three Review Passes

Для каждой итерации использовать один и тот же каркас:

1. Зафиксировать basis (основание) текущего прохода: `initial-diff-review` или `re-review-after-fixes`.
2. Зафиксировать, какие findings (замечания) входят в текущий проход.
3. Если это не первый diff-only проход, внести и верифицировать исправления.
4. Передать текущий результат свежему субагенту на критическое review.
5. Сразу после ответа субагента записать `reviews/iteration-N.json` через `scripts/persist_review_pass.py`.
6. Разобрать замечания субагента.
7. Поднять бизнес-вопросы пользователю, если они появились.
8. Обновить `session-state.json` и оценить, нужен ли следующий проход.

Останавливать цикл раньше третьего прохода, если одновременно выполняются все условия:

- последний проход не нашёл ни одного открытого finding любого уровня, которое требует кодового изменения, теста, миграции, перестройки контракта или явного бизнес-решения;
- если в сессии review-субагента был доступен `$final-gate-review`, этот skill был использован и тоже не дал новых findings;
- у субагента не появилось новых business questions (бизнес-вопросов) к пользователю;
- если в рамках текущего прогона были кодовые правки, текущий контур прошёл обязательные inspections (инспекции), quality checks (проверки качества) и нужные тесты по локальным правилам;
- если прогон остался review-only и кодовые правки не вносились, зафиксировано, что цикл завершился на нулевом пакете исправлений после initial diff review;
- в текущем change set нет даже слабых открытых замечаний, которые могут быть устранены в рамках текущего контура;
- остаточный риск либо отсутствует, либо относится только к внешней непроверяемой среде вне текущего diff и явно не ослабляет merge readiness (готовность к вливанию) для кода.

## Review Pass JSON Contract

Каждый `reviews/iteration-N.json` должен содержать минимум такие блоки:

```json
{
  "schema_version": 3,
  "iteration": 1,
  "mode": "re-review-after-fixes",
  "basis": {
    "review_surface": "staged diff",
    "base_branch": null,
    "notes": []
  },
  "in_scope": [],
  "fixes_applied": [],
  "verification": {
    "inspections": [],
    "quality_checks": [],
    "tests": []
  },
  "subagent": {
    "reviewer_skill": "final-gate-review",
    "prompt_summary": "",
    "raw_response": ""
  },
  "normalized_review": {
    "overall_summary": "",
    "remaining_issues": [],
    "regression_risks": [],
    "questions_for_user": [],
    "next_fix_batch": [],
    "what_was_checked": []
  },
  "user_answers": [],
  "exit_decision": {
    "decision": "continue",
    "reason": ""
  },

```

`raw_response` хранит исходный ответ субагента как есть. `normalized_review` хранит разложенную структуру для машинной обработки.

## Session State JSON Contract

`session-state.json` должен держать:

- metadata (метаданные) прогона;
- input (вход): task, raw findings, scope, project context;
- current status (текущий статус);
- confirmed business decisions (подтверждённые бизнес-решения);
- open questions (открытые вопросы);
- analysis (разбор замечаний и стратегия);
- working state (текущий пакет правок, changed surface, verification summary);
- final result (итог, resolved findings, remaining findings, residual risk).

## Subagent Contract

Субагенту давать роль критика остаточного риска, а не автора бизнес-решения.

Требовать от него:

- если в его сессии доступен `$final-gate-review`, использовать этот skill как основной режим критического ревью;
- если `$final-gate-review` недоступен, воспроизвести тот же стандарт строгости вручную;
- если текущий проход помечен как `initial-diff-review`, ревьюить исходный diff относительно base branch и не требовать предварительных кодовых правок до первого вывода;
- если изменение затрагивает существующий URL, список, историю, отчёт или иную внешнюю выборку, считать любое изменение состава выдачи или правил видимости потенциальным бизнес-решением, а не чисто техническим фиксом;
- если предлагается скрыть, фильтровать или перестать возвращать часть записей на существующем контракте, поднимать это как `Questions for user`, если только пользователь уже не подтвердил именно это поведение явно в рамках текущего прогона;
- проверять, какие findings реально закрыты, а какие замаскированы частичной правкой;
- искать регрессии, architectural drift (архитектурный дрейф), пропущенные проверки и слишком узкие фиксы;
- отдельно формулировать вопросы, которые требуют решения пользователя;
- не выбирать за пользователя поведенческий компромисс, scope фикса или допустимый технический долг;
- проводить анализ самостоятельно внутри своей сессии, без каскадной делегации и без запуска дочерних субагентов любым способом;
- возвращать вывод компактно и без воды.

После человекочитаемого ответа субагент обязан добавить fenced code block `json` с кратким машинно-читаемым summary (сводным итогом).

Использовать такую форму:

```json
{
  "overall_summary": "",
  "remaining_issues": [
    {
      "severity": "high",
      "summary": "",
      "details": "",
      "locations": [],
      "why_it_matters": "",
      "suggested_fix": ""
    }
  ],
  "regression_risks": [
    {
      "summary": "",
      "details": ""
    }
  ],
  "questions_for_user": [
    {
      "question": "",
      "why_it_matters": ""
    }
  ],
  "next_fix_batch": [
    {
      "step": "",
      "why": ""
    }
  ],
  "what_was_checked": [
    {
      "item": ""
    }
  ]
}
```

Если часть разделов пуста, возвращать пустой массив, а не пропускать ключ.

## Verification Rules

Считать верификацию обязательной частью каждого meaningful fix batch (содержательного пакета исправлений).

- Не запускать fix-oriented verification до первого review pass в `initial-diff-review`, если код ещё не менялся в рамках текущего прогона.
- Сначала запускать inspections (инспекции) по изменённому контуру, если этого требуют локальные правила.
- Затем запускать quality gate (контроль качества) по изменённому контуру.
- Затем запускать тесты в том объёме, который требует проект.
- Если локальные правила требуют полный прогон тестов в конце задачи, выполнять его до финализации.
- Сохранять результат верификации и в `reviews/iteration-N.json`, и в итоговом блоке `session-state.json`.

## Output Format

Итоговый ответ строить в таком порядке:

1. Коротко сформулировать, что и зачем исправлено.
2. Показать статус замечаний: закрыто, частично, осталось.
3. Показать подтверждённые пользователем бизнес-решения.
4. Показать остаточный риск и открытые вопросы, если они ещё влияют на выпуск.

Если пользователь просит только закрыть замечания без подробного отчёта, оставлять основным результатом статус замечаний и короткий блок с рисками.

## Rules

- Начинать с контекста и проверки замечаний по фактам.
- Всегда создавать и вести artifact workspace для каждого нового прогона навыка.
- Держать `session-state.json` главным mutable документом, а `event-log.jsonl` главным append-only журналом.
- Не исправлять ревью вслепую только по тексту комментария.
- Задавать вопросы по ходу работы, а не копить их до самого конца.
- Сначала определять entry mode, и на чистой ветке с base branch начинать с initial diff review до любых кодовых правок.
- Проводить от одного до трёх независимых review passes разными свежими субагентами.
- Субагент, запущенный в рамках этого skill, проводит review сам и не запускает других субагентов никаким способом.
- Любое изменение состава выдачи на существующем контракте считать неподтверждённой бизнес-развилкой, пока пользователь явно не согласовал именно это изменение.
- Любой вопрос, влияющий на бизнес, обязан быть задан пользователю; молчаливое допущение запрещено.
- Если в сессии review-субагента доступен `$final-gate-review`, его нужно использовать.
- Не начинать fix batch до первого review pass, если выбран `initial-diff-review`.
- Не считать cycle complete (цикл завершённым), пока остаётся хотя бы одно открытое замечание любого уровня.
- Не маскировать слабые findings как residual risk ради завершения цикла.
- Не выдавать предположения за подтверждённые факты.
- Не принимать бизнес-решения за пользователя.
- Не порождать россыпь файлов на каждый подпункт. Один review pass хранить в одном JSON-файле.
- После каждого `wait_agent` сначала сохранять raw response и normalized review, потом продолжать работу.

## Trigger Examples

- `Используй $iterative-review-fix и закрой эти замечания ревью с повторной проверкой результата.`
- `Разбери review findings, исправь подтверждённые проблемы и прогони re-review до устойчивого состояния.`
- `Доведи этот change set до merge-ready через цикл review -> fix -> re-review, а спорные бизнес-моменты сначала согласуй со мной.`
